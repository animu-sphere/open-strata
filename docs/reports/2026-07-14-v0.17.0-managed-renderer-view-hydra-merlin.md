# v0.17.0 managed renderer view acceptance with hdMerlin

## Scope

This report records the first downstream acceptance run of the v0.17.0
renderer lifecycle against hdMerlin. The run covered the no-argument managed
view path, an installed Hydra product, downstream host tests, external build
validation, and renderer report conflict handling.

This is observed evidence, not a replacement for the renderer contracts in
the architecture and reference documentation.

## Tested identities

| Component | Identity |
| --- | --- |
| OpenStrata | `ost 0.17.0`, repository commit `030c119` |
| Downstream project | hdMerlin `0.3.0`, commit `70c8c9f035d79844b2e6075ae275f22a07ebe03a` |
| Host | Windows x86_64, MSVC 19.34.31937, toolset 14.34.31933 |
| Runtime | `cy2026-windows-x86_64-py313-usd` |
| Runtime digest | `acebd2e48f70318911d4aac3b410149982c41a91e1bdc355c03aa5378d952e02` |
| GPU path | AMD Radeon Graphics, Vulkan loader 1.4.350, API 1.4.315 |

The hdMerlin branch was clean before and after the run. The adoption manifest
used its `VERSION` file as the version source and resolved all five declared
target mappings.

## Managed view result

Running `ost renderer view` with no runtime, build-directory, install-prefix,
or scene arguments selected the single compatible real `usd` runtime. The
default Ninja configure and build completed successfully, then the adopted
`MERLIN_ENABLE_HYDRA2` fallback selected and built the shared Hydra product.

- The initial core/Vulkan build compiled 22 targets and completed the 1,000
  frame 64 x 64 headless run with Vulkan validation enabled.
- The Hydra build compiled eight targets, including `hdMerlin.dll`, discovery,
  and render-buffer tests.
- The atomic completion record was written to
  `build/cy2026-windows-x86_64-py313-usd/.ost-build-complete.json` with the
  runtime identity and digest, host compiler, generator, Release configuration,
  `renderer-hydra2` intent, and adopted cache settings.
- The product was installed under
  `.strata/renderer-view/hdMerlin/release`, including the plugin library,
  resources, shaders, smoke scene, and Python packages.
- `usdview` launched with `renderer=Merlin` and the installed smoke scene and
  remained alive and responsive until the acceptance process was stopped.

`ost --json validate --profile usd` passed the atomic build, renderer manifest,
renderer evidence, core/backend/GPU, validation, color, depth, persistence, and
runtime-compatibility checks. Install-tree and Hydra host assertions remained
explained `SKIP`; the current canonical report does not infer those assertions
from a process launch or from project-owned CTest results.

## Hydra host tests

The installed downstream suite was run independently:

```text
ctest --test-dir build\cy2026-windows-x86_64-py313-usd -C Release \
  -R "^merlin-hydra2-(discovery|render-buffer|usdview-smoke)$" \
  --output-on-failure
```

All three tests passed:

| Test | Result | Evidence covered |
| --- | --- | --- |
| `merlin-hydra2-discovery` | PASS | plugin registration, render-delegate creation |
| `merlin-hydra2-render-buffer` | PASS | color, depth, and ID CPU-buffer behavior |
| `merlin-hydra2-usdview-smoke` | PASS | first frame and stable scene updates |

The headless usdview test exercised baseline, points, topology, transform,
visibility, camera, and resize phases. Every phase wrote four buffers, produced
covered pixels, kept Vulkan validation enabled, and reported zero validation
messages. It also recorded scene and resource revision changes and emitted seven
PNG images plus `merlin-regression.log` in the downstream build tree.

## External build and report merge checks

Validation against hdMerlin's real prebuilt directory with
`ost --json validate --profile usd --build-dir out-hydra` passed the
`external-build` check together with the renderer manifest/evidence and
core/Vulkan assertions. It correctly left OST-owned configured, built, and
runtime-compatible claims as `SKIP` because the external tree did not carry an
OST completion record.

Renderer report merging also preserved the v0.17.0 conflict policy:

- A duplicate `renderer.gpu.frame` overlay was rejected without `--replace`.
- The same overlay merged successfully with `--replace`.
- Replacing an existing `FAIL` with `PASS` was rejected even with
  `--replace`, and no output report was written.

## Follow-ups

- Define an explicit path for project-owned discovery, delegate, render-buffer,
  and host-smoke evidence to be emitted or merged into the canonical renderer
  report. Until then, the canonical host checks must remain explained `SKIP`.
- Repeat the core/Vulkan/Hydra acceptance on the hosted Linux, macOS, and other
  supported OpenUSD cells; this run proves only the Windows cy2026 cell.
- Reduce downstream CMake noise from intentionally supplied but unused
  `OST_RENDERER_ADAPTERS` and runtime provenance cache variables.
- The optional PyOpenGL NumPy warning was non-fatal in the headless test, but
  should be removed from a polished interactive environment.
