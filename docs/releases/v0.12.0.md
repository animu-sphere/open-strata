# v0.12.0 — hosted macOS CI + runtime-contract hardening

## Objective

Make the hosted-macOS CI and runtime contract first-class so generated source CI
runs on a clean GitHub-hosted runner without hand edits. The 2026-07-11 macOS
OpenUSD 26.05 PR-CI dogfood proved the public GHCR runtime pull, artifact
verification, and Windows source-CI path, but the hosted `macos-15-arm64` lane
only passed after two repo-local temporary repairs: `actions/setup-python` for
Python 3.13, and `chmod +x` on the materialized runtime `bin/`. v0.12.0 turns both
repairs into product behavior.

## Completed capabilities

- **Executable tool bits are preserved and verified end to end.** `runtime export`
  / artifact packaging read each staged file's real mode and pack an executable
  file as `0o755` (everything else stays canonical `0o644` for a deterministic
  archive), recording the runnable bit as `files[].executable` in the producer
  manifest; extraction restores the mode; `walk_archive` / `verify` carry and
  compare `executable`, so a tool that lost `+x` is a per-file manifest mismatch
  rather than a silent pass. `runtime validate` gained a Unix-only
  `bin-tools-executable` check that fails any real runtime whose top-level `bin/`
  tools are not executable, and `runtime export` re-runs validation before packing
  so a stale `validation: passed` manifest cannot publish a non-runnable runtime.
- **Schema-generation Python is an explicit runtime/CI contract.** A source cell
  declares the CPython ABI its schema tooling needs with `host_python: "3.13"`
  (validated `major.minor`, source-lane-only); `ost ci generate github` renders a
  SHA-pinned `actions/setup-python` for exactly that ABI on hosted runners before
  `ost plugin build`, recording the resolved Python source as
  `.ost-ci/python-setup.json`. Self-hosted runners keep their operator Python (the
  step is gated on `matrix.hosted`). Local builds already fail before
  `usdGenSchema` with a precondition naming every interpreter candidate.
- **Pre-build prerequisites are modeled in the CI graph, not scripted.** Modeled
  prerequisites (an always-on `ost runtime validate` of the freshly materialized
  tree, teed to `.ost-ci/runtime-validate.json`, plus the hosted `host_python`
  setup) render in a first-class section between runtime materialization and the
  build — distinct from the post-pyramid `source_checks`. No broad arbitrary
  pre-build shell hook was added.
- **CI evidence for repaired assumptions.** The source lane writes
  `.ost-ci/runtime-validate.json` (runtime-bin mode validation) and
  `.ost-ci/python-setup.json` (Python ABI + resolved source) alongside the existing
  runtime-artifact / OCI-digest evidence, all uploaded with the report.

## Compatibility notes

- No CLI or manifest breaking changes. `host_python` is a new optional
  source-cell field in `openstrata.ci.yaml`; existing matrices without it render
  as before (no hosted `setup-python` step).
- The archive format now records a per-file `executable` bit. Artifacts produced
  by v0.12.0 remain readable by earlier consumers, but only v0.12.0+ enforces and
  restores the runnable-tools invariant; regenerate/republish runtimes to gain it.
- Linux x86_64 is the first-class target; macOS arm64 and Windows are modeled and
  exercised in CI. This release's dogfood ran on `macos-15-arm64`, `windows-2022`,
  and GHCR.

## Known limitations / remaining follow-ups

These are the parts of the v0.12.0 milestone that need a Mac + live GHCR and are
tracked as active follow-ups in the [roadmap](../roadmap/README.md):

- Republish the public cy2026 macOS arm64 OpenUSD 26.05 SDK with preserved
  executable bits and confirm a clean GitHub-hosted `macos-15-arm64` source-CI
  lane reaches `ost plugin test --up-to 5` with no `chmod` repair.
- Remove the temporary repo-local `actions/setup-python` and `chmod` repairs in
  the downstream fixture repo after the runtime is republished and its CI is
  regenerated from `openstrata.ci.yaml`.
