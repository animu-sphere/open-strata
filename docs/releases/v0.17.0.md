# v0.17.0 - renderer build truth + managed Hydra inspection

## Objective

Close the lifecycle gap exposed by renderer dogfooding. v0.16.0 added renderer
and Hydra contracts; v0.17.0 makes the build state fail closed, makes external
CMake failures bounded and diagnosable, and turns `ost renderer view` into the
normal configure -> build -> install -> usdview loop without inventing a second
renderer build lifecycle.

## Completed capabilities

- **Atomic build-completion evidence.** `ost build` no longer treats a build
  directory as proof of completion. Tool-owned builds write a completion record
  only after configure, build, and output verification succeed; the record is
  bound to the target id, runtime digest, compiler fingerprint, generator,
  source project, build directory, and effective build intent. Starting a new
  build invalidates the previous record first, so interrupted builds, stale
  CMake caches, or copied reports cannot pass as fresh OST build evidence.
- **Recoverable external tool execution.** Configure/build progress now reports
  the active phase, elapsed time, child PID, log path, and bounded output tail.
  Configure and build timeouts are configurable, cancellation cleans up process
  trees, Ninja remains the default, and a generator override is available for
  projects that need it. Dry-run and execution share one generated-file plan,
  and the default managed preset output is the tool-owned
  `CMakeUserPresets.json`.
- **Managed one-command Hydra inspection.** `ost renderer view [scene]` resolves
  a compatible pulled real OpenUSD runtime, requests the shared `hydra2` build
  intent, incrementally builds a fingerprinted managed tree, stages the install
  tree, discovers the installed Hydra plugin metadata, selects the renderer in
  usdview, and launches the session with an isolated runtime environment.
- **Honest manual build escape hatch.** `--build-dir` remains supported for an
  external or prebuilt CMake tree, but that path is validated and labeled as
  manual evidence. OST may install and inspect it; it does not rebuild it or
  claim `ost build` produced it.
- **Renderer adoption and evidence transport.** `ost renderer adopt` starts as a
  read-only plan, writes manifests only with `--write`, preserves an existing
  authoritative version source through `--version-file`, and records adopted
  provenance rather than generated-scaffold provenance. Independent renderer
  reports merge under explicit conflict rules: duplicate assertion ids require
  replacement, and FAIL results cannot be silently downgraded.
- **Runtime and packaging correctness.** Managed renderer builds record the
  selected OpenUSD runtime id, prefix, and digest through CMake cache variables
  so launch-time checks can catch mismatched runtime discovery. Multi-config
  packaging and renderer runtime checks were tightened around the same evidence
  model.
- **Generated support references and portable Linux guidance.** Supported
  `OST_*`, generated CI, GitHub variable, and CMake-cache variables now come
  from one structured source and regenerate the environment-variable reference.
  Platform support data is likewise structured for generated documentation, and
  the portable Linux runtime guide plus reference container recipe document the
  glibc-floor constraints for release-quality Linux runtimes.

## Compatibility notes

- Tool-managed builds must now have a valid completion record. A directory that
  merely exists, an interrupted configure/build, or copied renderer evidence is
  rejected by validation until rebuilt through OST or supplied as explicit
  external evidence.
- The default generated preset file for managed builds is
  `CMakeUserPresets.json`. Scripts that assumed OST writes the project-root
  `CMakePresets.json` should invoke the generated user preset or install root
  presets deliberately.
- `OST_RENDERER_ADAPTERS=hydra2` is the shared project-facing build intent.
  Renderer projects still map that intent to their own CMake option and target
  graph; OST does not serialize private project policy into the renderer
  manifest.
- `ost renderer view --build-dir` no longer implies a tool-owned build. Evidence
  and diagnostics identify it as manual/external by design.
- Runtime fingerprint checks are stricter. Builds that previously discovered one
  OpenUSD SDK during CMake and launched against another runtime can now fail
  early instead of producing ambiguous usdview behavior.

## Known limitations / remaining follow-ups

- Hosted, real OpenUSD adoption remains the acceptance path for hdMerlin and the
  generated renderer skeleton across Linux, macOS, and Windows. The lifecycle is
  shipped; broader hosted runtime evidence is still active dogfood.
- Renderer manifests and reports remain `v1alpha1`; strict unknown-field
  rejection is intentional while the contract is applied to additional renderer
  projects.
- DCC host discovery and compatibility matrices move to v0.18.0. They must
  consume the renderer identity, installed Hydra product, runtime fingerprint,
  and evidence chain shipped here rather than introduce a parallel DCC renderer
  contract.
- Carry-over acceptance from v0.16.0 remains: the real `vrmContainer -> usdVrm`
  library producer/consumer, the hosted Windows/macOS/Linux `vrmSchema` L5
  checks, and the first downstream OIDC + live GHCR release-lane round trip.
- Explicit release-asset signing/Sigstore material and installer-side signature
  verification remain SEC-005 work; cargo-dist checksums and GitHub build
  provenance continue unchanged.

## Evidence

- Focused Rust unit and CLI integration tests cover completion-record creation
  and invalidation, managed build dry-runs, renderer view build intent and
  runtime fingerprint checks, external/manual build-dir labeling, renderer
  adoption plans, report merge conflict rules, generated support metadata, and
  multi-config package/runtime checks.
- Repository documentation now includes generated references for supported
  environment variables and platform support data, plus a portable Linux runtime
  guide and reference Dockerfile for glibc-floor-sensitive runtime builds.
- Local release gate on Windows passed: `cargo check --workspace`,
  `cargo fmt --all -- --check`,
  `cargo clippy --workspace --all-targets --locked -- -D warnings`,
  `cargo test --workspace --locked`, `python scripts/check_doc_links.py .`,
  `python scripts/check_docs_consistency.py .`, `cargo deny check`, and
  attribution regeneration/check.
