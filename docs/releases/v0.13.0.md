# v0.13.0 — release-quality packaging

## Objective

Close the release-quality boundary so a produced artifact is reproducible, lean,
and testable *from the package*. Dogfooding the clean-install / packaging path
(the downstream usd-vrm-plugins PR3 release lane) proved the consumer flow is
solid but surfaced the gaps that block a reproducible, distributable release:
packaging was not byte-reproducible and the shipped artifact still carried debug
symbols. v0.13.0 turns those gaps into product behavior and adds the primitives
to exercise a shipped package on a clean layout — the prerequisite for the trust
arc (v0.14.0+).

## Completed capabilities

- **Deterministic packaging honors `SOURCE_DATE_EPOCH`.** `ost plugin package`
  and `ost runtime export` stamp a fixed entry mtime on every archive entry
  (`PackOptions.mtime`, resolved from `SOURCE_DATE_EPOCH`) rather than the file's
  real mtime, so an unchanged staged tree repacks to a byte-identical digest.
  Ordering (sorted) and permissions (`0o755`/`0o644`) were already normalized;
  the archive is now a pure function of the staged files. The default mtime is
  `0`, byte-identical to the prior implicit zero, so existing digests are
  unchanged when `SOURCE_DATE_EPOCH` is unset. When it is set, the sidecar
  manifest's `created_unix` is pinned to it too, so the whole dist output
  reproduces — not just the archive.
- **Symbol-split artifacts ship lean by default.** `ost plugin package` splits
  debug-symbol sidecars (`.pdb`, `.dwo`) out of the main archive into a sibling
  `<name>-<version>-<target>-debug.tar.zst`, recorded under a `debug` key in
  `manifest.json` and as a second `SHA256SUMS` line. The lean main archive is
  byte-identical to a build with no symbols present. `--with-debug` keeps the
  symbols inside the main archive instead (a single fat package for local
  iteration). Local registry import/export and OCI push/pull preserve and
  verify the sibling archive together with the main artifact. Debug info
  embedded in an ELF/Mach-O binary is left in place
  (stripping it needs the toolchain, not a file move).
- **External-plugin-path run for clean-install testing.** `ost plugin run` gains
  `--plugin-path <dir>` (an installed/extracted plugin tree to put on the
  discovery path) and `--no-inject` (suppress the source bundle's build-tree
  `plugInfo`/`lib`/`python` paths). Together they point a session at the shipped
  layout instead of the build tree; `--no-inject` with no trees yields a bare
  "USD env only" runtime session (the `ost runtime run` shape). Session
  composition is generalized into `ost_plugin::session_env_from`.
- **Packaged-artifact smoke: `ost plugin test --from-package`.** Extracts the
  already-built package to a clean directory and runs the verification pyramid
  against the shipped tree, catching a build-tree path baked into
  `plugInfo`/`LibraryPath` that source-tree discovery cannot see (it walks the
  build tree, where the path still resolves). The digest-verify + unsafe-entry
  scan + unpack core is factored into a reusable `ost_artifact::extract_archive`.
- **`ost artifact extract --into <DEST>` alias.** `extract` accepts the
  destination as a named `--into <DEST>` flag in addition to the positional
  argument, so the verb reads naturally alongside the other artifact commands.

## Compatibility notes

- No breaking CLI or manifest changes. Packaging with `SOURCE_DATE_EPOCH` unset
  produces byte-identical archives to v0.12.0 (the entry mtime default is
  unchanged). New `manifest.json` keys (`debug`) and `SHA256SUMS` lines appear
  only when debug sidecars are split out.
- The default `ost plugin package` now ships lean: a `.pdb`/`.dwo` that a prior
  version folded into the main archive is split into a sibling `*-debug` package.
  Pass `--with-debug` to restore the single-fat-package behavior.

## Known limitations / remaining follow-ups

- Symbol splitting is file-level: it separates `.pdb`/`.dwo` sidecars but does
  not strip debug info embedded in an ELF/Mach-O binary (that needs a toolchain
  `strip`/`objcopy` pass, not a file move).
- End-to-end `--from-package` on the downstream usd-vrm-plugins PR3 release lane
  is the intended dogfood; carry-over follow-ups from v0.12.0 (macOS runtime
  republish, GHCR push round-trip) remain tracked in the
  [roadmap](../roadmap/current.md).

## Evidence

- Verified end-to-end against the `_ost_runner_test/plugins/toy` bundle on a real
  `build`-source cy2026 Windows runtime: deterministic repack (byte-identical
  digest across runs), `SOURCE_DATE_EPOCH` honored (distinct digest + pinned
  `created_unix`), lean-vs-`--with-debug` split, `--plugin-path`/`--no-inject`
  session composition, and `plugin test --from-package` extract-and-verify.
