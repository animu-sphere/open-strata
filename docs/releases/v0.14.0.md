## Objective

Close the publish-side trust boundary on top of the v0.13.0 release-quality
artifacts. v0.13.0 made packages reproducible, lean, and testable from the
shipped layout; v0.14.0 adds a strict policy document, publisher identity
matching, and minimum-trust verification so a protected namespace can reject an
untrusted publish before any registry mutation.

## Completed capabilities

- **Strict artifact policy schema.** `openstrata-artifact-policy.toml` now models
  `minimum_trust`, protected registry/repository namespaces, and named allowed
  publishers. The parser is fail-closed with recursive unknown-field rejection,
  duplicate checks, schema-version checks, namespace validation, dangling
  publisher-reference checks, and stable `ARTIFACT_POLICY_*` error codes. Trust
  levels are ordered `local` < `unsigned` < `attested` < `verified` < `trusted`.
- **Protected publisher enforcement for OCI pushes.** `ost artifact push`
  discovers the nearest policy file (or accepts `--policy <FILE>`), selects the
  most-specific protected namespace, requests a GitHub Actions OIDC identity for
  the fixed `openstrata-artifact-publish` audience, and matches repository,
  workflow path, git ref, actor, and event against the namespace's allowed
  publishers before contacting the registry. `--allow-untrusted-publisher` is an
  explicit break-glass override and is recorded in human/JSON output.
- **Policy-aware artifact verification.** `ost artifact verify --policy <FILE>`
  runs the existing archive/per-file integrity checks and then compares the
  artifact record's trust against the policy minimum. Human output reports the
  actual and required trust levels; JSON output includes the policy path,
  minimum trust, pass/fail state, and policy error code.
- **Release-lane dogfood hardening.** `openstrata.ci.yaml` is strict at every
  authored object, so misspelled verification/publication controls fail instead
  of disappearing from generated workflow YAML. The GitHub bootstrap now writes a
  native Windows directory to `GITHUB_PATH`, records the exported path and
  executable, and verifies hosted Windows availability through native
  `ost.exe --version`. `ost plugin run -- python ...` and `python3` now resolve
  through the runtime-aware Python resolver, selecting a concrete interpreter
  matching the runtime ABI or failing before launch with an actionable
  precondition.

## Compatibility notes

- Existing projects without `openstrata-artifact-policy.toml` keep the previous
  unprotected `ost artifact push` behavior. A policy file only affects protected
  namespaces it declares.
- Artifact records now carry a `trust` value. Records written before this field
  deserialize as `local`; artifacts registered through the publish gate are
  `unsigned` until later provenance/SBOM work can raise them.
- Bare `python` / `python3` under `ost plugin run` is no longer resolved by the
  host `PATH` accidentally. Explicit executable paths remain explicit.

## Known limitations / remaining follow-ups

- v0.14.0 verifies publisher identity and minimum trust, but does not yet attach
  or verify per-artifact SBOM/provenance evidence. That is the v0.15.0 slice.
- Generated trusted release lanes remain future work. The v0.14.0 hardening keeps
  existing source-CI lanes strict and runnable, but the tag-triggered typed
  release workflow waits for trust plus provenance foundations.
- The bootstrap pin maintenance helper (`ost ci pin bootstrap --version ...`) is
  still a non-blocking follow-up; CI manifests can already hold exact per-target
  checksums.

## Evidence

- Local release gate on Windows: `cargo fmt --all -- --check`,
  `cargo clippy --workspace --all-targets --locked -- -D warnings`,
  `cargo test --workspace --locked`, `python scripts/check_doc_links.py .`, and
  `cargo deny check` passed on the prepare branch.
- Focused v0.14.0 hardening checks: `cargo test -p ost-build run_python --locked`
  and `cargo test -p ost-cli plugin_run_ --locked` passed after the
  `ost plugin run` ABI-resolution fix.
