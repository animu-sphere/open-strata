# v0.16.0 — dogfood substrates + generated trusted release CI

## Objective

Close the next composition and publication boundaries exposed by real plugin and
renderer dogfood. v0.15.0 made source-workspace closures executable and carried
SBOM/provenance evidence with artifacts; v0.16.0 adds executable plain-library
closures, a renderer/Hydra bootstrap contract, and a typed tag-triggered workflow
that turns those evidence bundles into policy-gated release candidates and an
isolated protected publisher.

## Completed capabilities

- **Executable plain-library closures.** The new
  `openstrata.library/v1alpha1` descriptor gives non-plugin CMake libraries a
  versioned identity, package and exported target, transitive library edges, and
  runtime directories. Workspace validation rejects missing, duplicate,
  incompatible, malformed, or cyclic dependencies before CMake; build/install
  proceeds deepest-first into the target workspace prefix.
- **Library-aware sessions and packages.** Source test/run/view sessions receive
  materialized library runtime directories, inspect/test evidence records the
  selected descriptors and prefixes, and plugin packages carry the selected
  runtime directories plus closure evidence. The `cpp-library` scaffold emits a
  consumable config package and descriptor.
- **Renderer Slice A and Hydra bootstrap.** The new
  `openstrata.renderer/v1alpha1` and renderer-report contracts drive a
  project-level scaffold with host-neutral core, extraction seam, Vulkan
  capability target, deterministic offscreen triangle validation, headless
  product, install-tree CTest, and generic `ost validate` integration. Optional
  Hydra 2 coverage separates plugin discovery, delegate creation, CPU
  RenderBuffer, and isolated usdview first-frame/stable-update evidence;
  `ost renderer view` composes the matching installed runtime session.
- **Trust-aware generated source/support CI.** Matrix-level PR, main, and release
  floors compose with target trust. Generated consumers require digest, SBOM,
  provenance, and optional builder-policy verification while ordinary PR/main
  jobs remain read-only and publish-free.
- **Typed trusted-release generation.** A strict additive `release:` contract
  selects `main` cells marked `publish: candidate`, requires an exact
  `v<version>` ref and matching bundle version, exact hosted bootstrap checksums,
  constrained repository checks, package-twice reproducibility, clean
  `--from-package` testing, immutable candidate handoff, and either draft or
  publish behavior. Generation writes `.github/workflows/ost-release.yml`.
- **Permission-separated publishing.** Candidate builders keep read-only
  repository permissions and cannot push. Only the final job receives GitHub
  OIDC and package permissions, re-imports and re-verifies every candidate in a
  fresh store, authorizes the live publisher identity, and pushes to the typed
  OCI destination.
- **Evidence-derived handoff trust.** `ost artifact verify` now reports stored,
  evidence-derived, and effective trust separately. Valid subject-bound
  provenance establishes `attested`; matching an allowed publisher together
  with a valid SBOM applies that publisher's policy trust. The derivation is
  verification-local, so copying/importing `record.json` never grants sticky
  trust.
- **Windows Level-5 transport correctness.** USDA flattening captures through a
  temporary output file instead of a Windows text-mode stdout pipe, preserves
  semantic carriage returns inside authored strings, and distinguishes a real
  authored/generated mismatch from transport translation.

## Compatibility notes

- Support-matrix schema 1 remains additive. Matrices without `release:` and
  without `publish: candidate` keep their prior source/support behavior.
- A publish candidate is now fail-closed: it must be a `main` source cell, meet
  `release_min_trust` (at least `verified`), have a matrix-level `release:`
  block and `trust.policy`, and use exact bootstrap checksums when hosted.
- `release.mode: draft` produces verified immutable handoffs without registry
  mutation. `publish` additionally requires a named publisher runner, an OCI
  repository without a mutable tag/digest, and optional protected environment.
- Library and renderer manifests are `v1alpha1`; strict unknown-field rejection
  is intentional while those contracts gather cross-project evidence.
- Downstream generated release lanes must bootstrap an `ost` version that
  understands this contract (v0.16.0 or newer) and pin the exact release-asset
  checksum for every hosted target they use.

## Known limitations / remaining follow-ups

- The real `vrmContainer -> usdVrm` library producer/consumer, the temporarily
  capped Windows `vrmSchema` L5 cell, and renderer core/Vulkan/Hydra adoption
  still need hosted Windows/macOS/Linux evidence. Their product contracts are in
  this release; the environment-dependent acceptance remains active dogfood.
- The generated publish workflow has deterministic YAML and CLI integration
  coverage, but its first downstream OIDC + live GHCR round trip remains an
  external release acceptance item. Existing GHCR visibility and credential
  constraints still apply.
- Product-level digest-pinned composition of multiple packaged bundle artifacts
  remains separate from source-workspace and library closure.
- Explicit release-asset signing/Sigstore material and installer-side signature
  verification remain SEC-005 work; cargo-dist checksums and GitHub build
  provenance continue unchanged.
- DCC host discovery and compatibility matrices are the v0.17.0 milestone.

## Evidence

- Focused Rust unit and CLI integration tests cover strict release-manifest
  validation, exact-pin enforcement, deterministic parseable workflow output,
  candidate/publisher permission separation, reproducibility/from-package gates,
  billing acknowledgement, and evidence-derived effective trust after a fresh
  import.
- Local release gate on Windows passed: `cargo fmt --all -- --check`,
  `cargo clippy --workspace --all-targets --locked -- -D warnings`,
  `cargo test --workspace --locked`, `python scripts/check_doc_links.py .`,
  `python scripts/check_docs_consistency.py .`, `cargo deny check`, and
  attribution regeneration/check.
