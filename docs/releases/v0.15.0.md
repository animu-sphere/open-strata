# v0.15.0 — workspace composition + artifact evidence bundles

## Objective

Make multi-bundle source workspaces executable from their authored dependency
graph and turn produced artifacts into evidence bundles. v0.14.0 protected
artifact publication with policy and publisher identity; v0.15.0 carries that
trust chain into the artifact record itself by preserving SBOM and provenance
sidecars through local storage, transport, and verification.

## Completed capabilities

- **Source-workspace dependency composition.** `requires.bundles` is now consumed
  after validation instead of being documentation-only. `ost plugin test
  --workspace` tests every bundle with its transitive dependency closure, while a
  selected `plugin doctor|test|run <bundle>` resolves the same closure when its
  containing workspace is unambiguous. Explicit `--with` remains an additive
  escape hatch for external or ad-hoc bundles.
- **Deterministic dependency builds.** `ost plugin build <bundle>` builds source
  dependencies in topological order, installs them into an OST-owned
  target-specific private prefix, and exposes that prefix through normal CMake
  package discovery. Generated source-CI cells consume the same manifest closure
  selected by `bundle:` without a second cell-level dependency list.
- **Fail-closed versioned manifest contracts.** Versioned plugin manifests reject
  unknown keys below `requires:`, require explicit schema contracts for
  schema-consuming bundles, preserve dependency-direction checks, and reserve
  `requires.libraries` with a stable failure until portable non-plugin library
  identity and discovery have their own contract.
- **Artifact evidence sidecars.** Plugin, project-package, and runtime producers
  generate SPDX 2.3 SBOM evidence, and producers with complete build metadata add
  SLSA/in-toto provenance. `record.json`, `SHA256SUMS`, local store
  import/export, pull, checksum, human output, JSON output, and OCI push/fetch
  preserve both `sbom.spdx.json` and `provenance.intoto.jsonl` as digest-checked
  sidecars.
- **Required-evidence verification.** `ost artifact verify --require-sbom` and
  `--require-provenance` enforce evidence presence and semantic binding. SBOMs
  must identify the OpenStrata artifact digest; provenance must bind the artifact
  subject, source repository/revision, and, when a policy is supplied, an allowed
  builder identity. JSON output reports required/present/pass state, evidence
  digests, matched publisher id, and stable `ARTIFACT_*` error codes.
- **Workspace dogfood hardening.** The schema CMake consumer now avoids MSVC
  `min`/`max` macro collisions and does not re-enter `pxrConfig` once `pxr` is
  already resolved. Level-5 USDA comparison keeps embedded string CRLFs semantic,
  emits an actionable line-ending hint when that is the only difference, and
  reports bounded diff evidence. Bundles that no longer own/co-host a schema stop
  rendering stale schema-source fragments.

## Compatibility notes

- Legacy plugin manifests without the versioned composition fields remain valid.
  Manifests that opt into `openstrata.plugin/v1alpha1` fail closed for unknown
  `requires:` keys, including the reserved `requires.libraries` field.
- Source-workspace composition does not make packaged plugin artifacts include
  their dependencies. Clean-install multi-package composition still needs a
  future product/artifact-closure descriptor that pins every member by digest.
- Normal `ost artifact verify` continues to accept older artifacts that do not
  carry SBOM or provenance evidence. Release and policy gates opt into mandatory
  sidecars with `--require-sbom` and `--require-provenance`.
- Provenance is emitted only when complete build metadata is available. A partial
  ambient CI environment is intentionally ignored instead of producing incomplete
  attestation data.

## Known limitations / remaining follow-ups

- Generated trusted CI remains the next release slice. v0.15.0 gives source-CI
  the right workspace closure and artifacts the right evidence, but does not yet
  generate protected runtime-publish or release-trust lanes from the support
  matrix.
- Runtime minimum-trust enforcement in `ost build` / `ost plugin test` remains a
  future policy hook. v0.15.0 verifies evidence and builder binding at artifact
  verification time.
- Product-level packaged workspace composition is still future work; the source
  workspace graph must not be used as a clean-install package graph.
- Live hosted acceptance for the downstream schema/file-format split and any
  live-registry round trip remain environment-dependent release evidence rather
  than a new CLI contract.

## Evidence

- Local release gate on Windows: `cargo fmt --all -- --check`,
  `cargo clippy --workspace --all-targets --locked -- -D warnings`,
  `cargo test --workspace --locked`, `python scripts/check_doc_links.py .`, and
  `cargo deny check` passed on the release-prep commit. `cargo check --workspace`
  refreshed `Cargo.lock` after the version bump.
- The merged implementation added focused Rust unit and CLI integration coverage
  for source-workspace closure, dependency build ordering, versioned manifest
  strictness, artifact evidence preservation, and required-evidence validation.
